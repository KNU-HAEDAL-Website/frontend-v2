구조와 의존 방향

기능을 추가할 때는 화면에서 서비스와 상태 계층으로 내려가는 방향을 유지하고, 공통 UI와 도메인 기능 UI의 책임을 구분합니다.

flowchart TD
  Main[main.tsx: StrictMode와 전역 CSS] --> App[App.tsx]
  App --> Provider[React Query Provider]
  App --> Toaster[전역 Toaster]
  Provider --> Router[Router와 route guards]
  Router --> Pages[pages: 화면 단위 조합]
  Pages --> Feature[components/feature: 도메인 UI]
  Pages --> Common[components/common·ui: 재사용 UI]
  Feature --> Service
  Feature --> Store
  Pages --> Service[service/api: API·query options]
  Service --> Config[service/config: Axios 인스턴스·인터셉터]
  Service --> Model[service/model: 생성된 API 모델]
  Pages --> Store[store: 영속 클라이언트 상태]
  Config --> Store

계층별 책임

영역책임새 기능을 추가할 때
pages라우트에 연결되는 화면과 화면 전용 조합새 화면의 진입점과 화면 전용 하위 컴포넌트를 둡니다.
components/uiRadix·CVA 기반의 디자인 시스템 프리미티브Button, Input, Dialog처럼 조합 가능한 기본 제어 요소를 추가합니다.
components/common앱 전반에서 재사용하는 의미 있는 조합 컴포넌트Logo, LinkButton, Pagination처럼 UI 프리미티브를 조합한 앱 공통 패턴을 추가합니다.
components/feature댓글, 게시글, 레이아웃 등 도메인 기능 UI여러 페이지에서 공유되는 도메인 UI를 둡니다.
service/api생성 API 클라이언트를 감싼 호출, Query 옵션, mutation용 함수API 경계와 Query 키를 도메인 모듈에 추가합니다.
service/configAxios 인스턴스와 인증 오류 처리보안·계약 영향이 있으므로 명시적 검토 없이 변경하지 않습니다.
storeZustand 기반 영속 클라이언트 상태서버 캐시가 아닌 UI·세션 상태만 추가합니다.
libQueryClient, Axios 생성 등 인프라 보조 코드여러 도메인이 공유하는 기반 동작에만 사용합니다.

components/common과 components/ui 구분

구분components/uicomponents/common
목적일관된 스타일·접근성을 제공하는 가장 작은 UI 단위여러 화면에서 반복되는 앱 수준의 사용자 경험을 캡슐화
예시Button, Input, Dialog, Select, SkeletonLogo, LinkButton, PageBreadcrumb, PaginationButtons, ImageInput
허용 의존성Radix, 스타일 유틸리티, 범용 props·접근성 상태components/ui, React Router 링크, 앱 공통 asset·유틸리티
두면 안 되는 것라우트, 도메인 모델, API 호출, Zustand 스토어특정 도메인의 API·스토어·권한 규칙
새 컴포넌트 판단다른 기능에서도 독립적으로 조합할 수 있는 기본 제어 요소인가?도메인과 무관하지만 앱에서 같은 흐름·표현이 반복되는가?

컴포넌트 의존성 기준

영역허용되는 의존성피해야 할 의존성
pages라우트 파라미터, 화면 조합, feature/common UI, API·상태 조율다른 페이지 구현의 직접 재사용
components/feature도메인 UI, API 조회·mutation, 필요한 세션 상태서로 무관한 도메인의 UI·화면 컴포넌트
components/uiprops와 범용 UI 상태도메인 API, Zustand 스토어, 특정 라우트에 대한 직접 의존
components/commoncomponents/ui, Router 링크, 공통 asset·유틸리티특정 도메인의 API·스토어·권한 규칙

함께 확인할 지원 영역

  • service/schema: Zod 기반 입력 검증
  • types: 역할·학기 등 앱 공통 타입
  • utils, constant: 역할 비교, 변환, 오류 메시지
  • data, assets: 정적 콘텐츠와 이미지·SVG
생성 코드: src/service/model/과 src/lib/http-client.ts는 Swagger 생성 파일입니다. 직접 수정하지 않고 호출부와 API 계약을 검토합니다.
